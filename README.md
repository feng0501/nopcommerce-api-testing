# nopCommerce 电商平台接口自动化测试

## 项目简介

本项目基于 **nopCommerce 4.60.6** 开源电商系统，借助浏览器抓包梳理前台各类业务接口，使用 **Postman** 搭建整套接口自动化测试脚本。测试范围涵盖用户注册、账号登录、商品搜索、购物车操作、优惠券使用、订单结算、账号登出等日常购物全流程。

项目重点解决了站点自带的 CSRF 验证问题，通过前置 GET 页面请求搭配脚本，实现动态抓取页面内 `__RequestVerificationToken`，并统一存入集合变量完成全局调用。

由于该平台的 CSRF Token 具备**单次使用即失效**特性，最开始所有 POST 请求共用同一个变量值，批量执行时仅首个 POST 请求能够正常通过，其余请求都会因令牌失效抛出 400 错误。后续调整优化执行顺序，在每一组业务 POST 请求前重新发起页面请求刷新最新令牌，替换更新集合变量，目前整套测试流程已经可以完整顺畅跑完，所有接口均可执行成功。

## 技术栈

- **测试工具**：Postman + Collection Runner
- **数据抓取**：浏览器开发者工具
- **令牌处理**：Postman Tests 脚本自动提取并更新 CSRF 令牌
- **结果输出**：HTML 格式测试报告、运行过程截图
- **代码托管**：GitHub

## 覆盖接口与测试点

| 模块 | 请求方法 | 接口路径 | 断言 / 校验点 |
|------|----------|----------|----------------|
| 刷新登录 Token | GET | `/login` | 页面正常加载状态码 200，成功提取登录专用令牌 |
| 用户登录 | POST | `/login` | 请求状态 302，自动重定向至网站首页 |
| 获取注册页 Token | GET | `/register` | 页面正常加载，提取注册操作所需令牌 |
| 用户注册 | POST | `/register` | 状态 302 跳转，后台数据库可新增用户信息 |
| 商品搜索 | GET | `/search?q=apple` | 接口返回 200，响应内容包含对应商品信息 |
| 获取加购 Token | GET | `/cart` | 进入购物车页面，刷新获取全新可用令牌 |
| 添加购物车 | POST | `/addproducttocart/catalog/4/1/1` | 状态 200 或 302，购物车内商品数量同步增加 |
| 获取购物车 | GET | `/cart` | 页面加载正常，展示已加入的购物商品 |
| 应用优惠券 | POST | `/cart` | 成功绑定优惠码，页面展示对应折扣金额 |
| 删除购物车商品 | POST | `/cart` | 正常移除指定商品，购物车列表同步更新 |
| 进入结账页 | GET | `/onepagecheckout` | 结算页面加载完成，重新提取下单令牌 |
| 保存配送地址 | POST | `/checkout/OpcSaveBilling/` | 地址信息提交成功，页面正常缓存信息 |
| 确认订单 | POST | `/checkout/OpcConfirmOrder/` | 订单提交完成，顺利完成下单流程 |
| 退出登录 | GET | `/logout` | 状态 302 跳转，清除本地登录会话 |

## 如何运行

### 1. 导入 Postman 集合与环境
- 下载仓库内 `collection.json` 测试集合文件与 `environment.json` 环境变量文件。
- 打开 Postman 软件，点击导入功能，上传两个文件完成导入配置。
- 右上角选中对应运行环境，核对基础访问地址 `baseUrl` 为 `http://localhost:5000`。

### 2. 准备测试环境
- 本地成功启动 nopCommerce 项目，配套数据库连接正常，可正常访问前台页面。
- 提前在网站后台创建可用优惠码，保证优惠券接口能够正常使用。
- 提前准备好测试账号，可直接使用现成账号，也可自行注册新账号使用。

### 3. 运行测试
- 打开建好的测试集合，点击集合右侧 Runner 批量运行按钮。
- 保持项目内预设的接口执行顺序不变，直接点击运行即可。
- 整套流程自动依次执行，无需手动干预，全部接口均可正常通过。

### 4. 生成测试报告
- 批量测试结束后，在运行结果页面导出 HTML 格式测试报告留存。
- 也可直接截取全部接口运行成功页面，作为项目测试成果凭证。

## 项目亮点

- **适配站点安全机制**：精准解决 CSRF 一次性令牌难题，摒弃手动复制令牌的繁琐操作。
- **贴合真实用户购物操作逻辑**：接口执行顺序完全还原日常线上购物步骤，实用性强。
- **统一使用集合变量管理令牌**：搭配分组刷新令牌逻辑，结构清晰便于后期修改维护。
- **项目移植性强**：仅需导入对应文件并修改本地访问地址，即可在其他设备直接运行。

## 调试过程遇到的问题与解决思路

| 出现问题 | 问题成因 | 对应解决方式 |
|----------|----------|----------------|
| 登录后添加购物车接口返回 400 | 未提前抓取页面令牌，请求缺少合法验证参数 | 在登录请求前新增页面 GET 请求，提前提取登录令牌 |
| 批量运行时后半段 POST 接口全部报错 | 令牌单次使用后直接失效，共用旧变量无法二次使用 | 拆分业务模块，每组 POST 业务前重新访问页面刷新全新令牌 |
| 改用购物车页面获取令牌后流程紊乱 | 新令牌覆盖原有变量，打乱后续结算流程 | 严格划分接口执行顺序，购物车、结账两大模块分开刷新令牌 |
| 部分商品页面路径访问 404 | 本地项目版本不支持简写商品访问路径 | 放弃专用商品页，改用购物车、搜索页这类通用页面提取令牌 |
| 批量运行丢失登录状态 | 批量运行模式下会话缓存未正常开启 | 开启 Postman 内置 Cookie 持久化功能，保持全程登录状态 |
